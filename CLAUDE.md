## Project Overview

TechDungeon is a 2D roguelike/dungeon crawler built with **Unity 6000.3.8f1** (C#). It uses **URP 2D** rendering, the **New Input System**, and **A\* Pathfinding** for enemy navigation.

## Build & Test

This is a Unity project — there are no CLI build commands. All building, running, and testing happens through the Unity Editor.

- **Tests**: Run via Unity Test Runner (Window > General > Test Runner). Tests are in `Assets/Tests/Editor/` using NUnit.
- **MCP Unity server**: Auto-starts on port 8090 for IDE integration.
- The `.csproj` and `.sln` files are auto-generated by Unity — do not edit them manually.

## Architecture

### Service Locator (Dependency Injection)
`Assets/Scripts/ServiceLocator/` — Central DI framework used throughout the project. Services register themselves (e.g., `ObjectPool`, `PlayerMovementController`) and are retrieved via `ServiceLocator.For(this).Get<T>()`. Supports both global and scene-scoped service lifetimes.

### Object Pooling
`Assets/Scripts/Gameplay/ObjectPool/ObjectPool.cs` — Reuses GameObjects (enemies, projectiles) to avoid runtime allocations. Objects are returned to the pool after a configurable duration. Registered as a global service.

### Enemy AI (State Machine)
`Assets/Scripts/Gameplay/Entity/Enemy/` — Enemies use an `EnemyStateMachine` with `IState` interface (`IdleState`, `SeekState`, `AttackState`). `EnemyController` orchestrates all enemy subsystems (movement, shooting, targeting, health, animation).

### Entity System
`Assets/Scripts/Gameplay/Entity/` — Base `Entity` class with `EntityHealth` and `EntityAnimationController`. Both player and enemies build on this.

### Player System
`Assets/Scripts/Gameplay/PlayerObject/` — `PlayerMovementController` handles input-driven movement with roll mechanics. Weapon system uses `IWeapon` interface with `Weapon`, `WeaponShooting`, and `WeaponHolder`.

### Input
`Assets/Scripts/Input/InputReader.cs` — ScriptableObject wrapping the New Input System. Exposes C# events (Move, Attack, Interact, Roll, Look). `InputSystem_Actions.cs` is auto-generated — do not edit.

### Loot & Interactables
`Assets/Scripts/Gameplay/Interactable/` — `Lootable` handles drop spawning with scale/position lerp animations. `BounceEffect` provides physics bounce. `Chest` is an `IInteractable` with configurable drops via `ChestSettings` ScriptableObject. Rarity system uses shader-based outline rendering (`_OutlineThickness`, `_OutlineColor`, `_PixelSize`).

### ScriptableObject Settings
`Assets/Scripts/Gameplay/Settings/` — Data-driven configuration: `PlayerSettings`, `EnemySettings`, `ProjectileSettings`, `ChestSettings`. Prefer ScriptableObjects over hardcoded values.

### Projectiles
`Assets/Scripts/Gameplay/Projectile/` — Velocity-based movement with collision damage, hit tracking, and automatic pool return after lifetime.

## Key Conventions

- **Interfaces**: `IInteractable`, `IWeapon`, `IInputReader`, `IState` — used for contracts and testability.
- **Component composition**: Systems are composed of small, focused MonoBehaviours rather than large monolithic scripts.
- **Pooling over Instantiate/Destroy**: Use `ObjectPool` for frequently created/destroyed objects.
- **Events/delegates**: Preferred for cross-system communication (see `InputReader` action events, `EntityHealth` callbacks).
- **2D Physics**: All physics use 2D variants (Rigidbody2D, Collider2D, etc.).
